## 2. 分析对象边界

### 2.1 包含什么


| 维度        | 范围                                                              |
| --------- | --------------------------------------------------------------- |
| **项目**    | P1–P11（见 [plan.md](./plan.md)）                                  |
| **技术面**   | C / `extern "C"` 的 Rust 绑定与封装（`-sys`、safe、async、**上层适配 crate**） |
| **C 头文件** | 通过 `wrapper.h` 等能接触到的**原始 C API 表面**（见 §4.4）                    |
| **产出**    | `docs/<路径>/FFI-ANALYSIS.md`，**12 个固定章节**（§0–§11，见 §5）           |


### 2.2 明确不包含（非目标）


| 非目标             | 说明                                                 |
| --------------- | -------------------------------------------------- |
| **跨项目对比**       | 单篇中不写「比 xxx 更好/更差」                                 |
| **修改样本源码**      | 样本树只读；产出仅在 `docs/`                                 |
| **C++ 全量绑定**    | 不系统梳理 C++ ABI                                      |
| **上游 C 库完整手册**  | 不重写 C API 参考                                       |
| **协议/业务语义深读**   | 上层适配 crate **要写做了什么/为何/如何做**，但不展开协议或业务细节           |
| **性能调优 / 安全审计** | 不系统化                                               |
| **构建/运行验证**     | **不做** `cargo build` / `cargo check` / 硬件实测（裁定 #1） |


### 2.3 与阶段 C / D 的衔接

- 阶段 C/D 可引用各篇 `FFI-ANALYSIS.md` 章节号。
- 单项目阶段不写规范性「应做/不应做」；**§10 架构决策推断**写**本项目为何如此**（推断），供阶段 D 归纳实践时作证据。

## 3. 分析深度与粒度

### 3.1 总体深度


| 级别          | 要求                                       |
| ----------- | ---------------------------------------- |
| **架构层**     | crate 分层、绑定方式、链接策略、错误/资源/并发模型            |
| **机制层**     | 代表性路径写清机制；证据以**路径 + 符号/模块名**为主（不要求行号）    |
| **API 目录级** | 不穷举公开 API；模块树 + 典型类型即可                   |
| **架构动机**    | **§10 专章**：结合 C 头文件与 Rust 分层，推断主要架构决策的原因 |


### 3.2 Workspace / 多 crate


| 规则        | 说明                                                  |
| --------- | --------------------------------------------------- |
| **一篇总文档** | 每个项目仍只产出一份 `FFI-ANALYSIS.md`                        |
| **章内分节**  | §1、§4 按 member crate 分节                             |
| **阅读顺序**  | 先 `-sys` / bindings → safe / async → **上层适配**（§7.2） |


### 3.3 证据强度


| 标记          | 含义                                       |
| ----------- | ---------------------------------------- |
| `confirmed` | README、注释、上游文档等明确写出                      |
| `inferred`  | 从 C 头文件、代码、build 推断 — **§10 推断结论默认用此标记** |
| `unknown`   | 材料不足 → 写入 §11「待澄清」                       |


### 3.4 证据呈现（简化，裁定 #4）


| 做法                                         | 不做              |
| ------------------------------------------ | --------------- |
| 正文与 §9 以 `**路径`** 或 `**路径` + 符号/模块名** 指向源码 | 不要求 `path:line` |
| §9 每条一两句话说明「该文件支撑什么结论」                     | 不做冗长摘录          |
| §9 规模约 **8–15 条**，按 crate 分组即可             | 不追求 30 条全覆盖     |

## 4. 源码阅读范围

### 4.1 必读（静态阅读，不构建）


| 顺序  | 路径模式                                   | 目的                         |
| --- | -------------------------------------- | -------------------------- |
| 1   | `README`*、`.github/workflows/*`（若有）    | 定位、CI                      |
| 2   | `Cargo.toml`（根及 member）                | workspace、features、`links` |
| 3   | `build.rs`、`wrapper*.h`、bindgen 配置     | 绑定与链接                      |
| 4   | `*-sys` / `bindings` / `generated*.rs` | FFI 表面                     |
| 5   | safe / 主 crate 核心模块                    | RAII、`unsafe` 边界           |
| 6   | `tests/`、`systest/`（若有）                | 测试策略                       |
| 7   | `examples/`                            | **简单阅读**（见 §4.3）           |


### 4.2 选读


| 路径                        | 深度                    |
| ------------------------- | --------------------- |
| `fuzz/`                   | 有则 §8 一句带过            |
| `maint/`、`scripts/`       | 绑定再生流程                |
| vendored C **实现**（`.c` 等） | 不读；只看 `build.rs` 如何编链 |
| `doc/` 中与 FFI 相关图         | 与代码交叉验证               |


### 4.3 Examples（裁定 #3：简单阅读）


| 做                             | 不做                |
| ----------------------------- | ----------------- |
| 列出 example 名称与入口文件            | 逐 example 详解业务逻辑  |
| 浏览主流程，判断是否涉及 **FFI 用法演示**     | 把 example 当必读深读材料 |
| 在 §8 用**简短列表**说明「有无、是否展示 FFI」 | 每个 example 写长摘要   |


### 4.4 原始 C 头文件（裁定 #5：可以读）


| 做                                                 | 不做                  |
| ------------------------------------------------- | ------------------- |
| 阅读 `wrapper.h` **及其 `#include` 到的 C 头**（系统头或仓库内头） | 通读 vendored `.c` 实现 |
| 用于理解 **C API 形态**（handle、回调、所有权提示等）               | 编写 C API 手册         |
| 作为 **§3、§6、§10** 的输入：C 侧约束 → Rust 侧设计动机           | 脱离 Rust 项目单独评价 C 库  |


**阅读过程中**：对照 C 头与 Rust 分层，随时记录疑点，最终收敛到 **§10 架构决策推断**。

### 4.5 明确不读

- 工作区外其他项目  
- Git 历史 / issue（除非 README 指向某决策）  
- 与 FFI 无关的 example 业务细节

## 5. 产出章节 ↔ 分析任务（`FFI-ANALYSIS.md`）

共 **12 章（§0–§11）**，与 [plan.md 三](./plan.md#三单项目分析模板统一产出) 一致。不适用写 `N/A — <原因>`，**不删节**。


| 章节               | 必须回答的问题                                             |
| ---------------- | --------------------------------------------------- |
| **0. 摘要**        | 绑谁、几层 crate、面向谁                                     |
| **1. 仓库结构**      | workspace 成员；tests；examples 清单（简表）                  |
| **2. 原生依赖**      | 系统库 / vendored / pkg-config / `links`               |
| **3. 绑定生成**      | bindgen / 手写 / 混合；再生流程；**C 头暴露的主要 API 形态**（来自 §4.4） |
| **4. 分层与公开 API** | 各层职责、模块树；**上层适配 crate 专节**（见 §7.2）                  |
| **5. 资源与生命周期**   | 分配/释放、`Drop`、顺序                                     |
| **6. 错误与安全边界**   | 错误类型、`unsafe` 契约；C 错误码/返回值习惯（若有）                    |
| **7. 并发与 async** | `Send`/`Sync`、线程模型、async 集成                         |
| **8. 测试与示例**     | 测试类型、硬件依赖；examples **简单阅读**结论                       |
| **9. 证据索引**      | **简化**：8–15 条，路径为主（§3.4）                            |
| **10. 架构决策推断**   | **为何**采用当前架构（见 §5.1）                                |
| **11. 待澄清**      | 本项目内未决问题                                            |


### 5.1 §10 架构决策推断（裁定 #7：新增）

本章回答：「**结合原始 C 头文件与本项目实际架构，推断作者为何这样设计？**」

**输入材料**

- C：`wrapper.h` 及所 include 的头（API 形态、资源模型、线程/回调约束等）
- Rust：crate 划分、`build.rs`、bindings 策略、safe/async/上层适配实现

**每条推断建议包含**


| 字段                | 说明                                         |
| ----------------- | ------------------------------------------ |
| **决策**            | 一句话（如「独立 `-sys` crate + safe 分层」）          |
| **观察到的 C 侧事实**    | 来自头文件/绑定的客观描述                              |
| **观察到的 Rust 侧事实** | 本项目实际结构                                    |
| **推断动机**          | 为何该 C 形态会导致或合理导致该 Rust 选择                  |
| **证据**            | 简化路径即可                                     |
| **置信度**           | `confirmed` / `inferred`（推断类默认 `inferred`） |


**建议覆盖的决策类型（有则写，无则 N/A）**

- crate 拆分（`-sys` 是否独立、workspace 成员）
- bindgen vs 手写、vendored vs 系统库
- opaque / newtype / 裸指针策略
- 错误映射方式
- 是否增加 async 层或上层适配 crate（**与 §7.2 呼应**）
- 测试 / mummy / mock 策略（若与架构相关）

**边界**

- 写「**推断**」，不写「应该」；不与其它样本项目对比。
- 无法推断时写入 §11，不在 §10 编造。

## 7. 按项目类型的范围差异

### 7.1 纯 `-sys`（P1、P8、部分 P10）

加强 §3、§6、§10（C 形态如何约束「薄封装」）；§4 若无 safe 则注明。

### 7.2 多 crate / 上层适配（P3、P6、P7）— 裁定 #2


| Crate 类型                                    | 范围                                                                                                   |
| ------------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| `*-sys` / bindings                          | 必纳                                                                                                   |
| 核心 safe                                     | 必纳                                                                                                   |
| errors / macros                             | 必纳                                                                                                   |
| **上层适配**（如 `rdma-io-tonic`、`rdma-io-quinn`） | **必纳专节**，须写清三点：**做了什么**、**为什么要做**、**怎么做的**（如何依赖下层 FFI、新增哪些类型/ trait / `unsafe`、如何把 C/RDMA 能力接到上层运行时） |
| bench / `bnd-rdma-gen` 等                    | §1 列出；仅生成绑定则并入 §3                                                                                    |


不展开协议语义；聚焦 **FFI 与架构边界**。

### 7.3 RDMA（P5–P11）

§4 / §7 / §10 中若有则写：verbs 覆盖面、async/CQ、硬件依赖、mock/mummy。

### 7.4 Vendored C（P4 等）

§2、§3 写 vendored 范围；**不读** vendored `.c`；§10 可推断「为何 vendored」。
