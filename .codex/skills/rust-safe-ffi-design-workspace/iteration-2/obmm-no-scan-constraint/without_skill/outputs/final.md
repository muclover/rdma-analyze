## 建议分层

建议采用两层 crate：

1. `obmm-sys`

   负责原始 FFI 绑定、链接配置和 ABI 边界声明。该层尽量薄，不提供 Rust 风格抽象，只表达 C ABI。

2. `obmm`

   负责安全封装、错误映射、生命周期管理、线程安全约束表达和符合 Rust 习惯的 API。

如果后续确认 obmm 只提供 C++ API，建议先增加一个很小的 C ABI shim，再由 `obmm-sys` 绑定该 shim；不要直接把复杂 C++ ABI 暴露给 Rust。

## `obmm-sys` 设计草案

`obmm-sys` 的职责：

- 声明 `extern "C"` 函数和不透明类型。
- 通过 `build.rs` 或 crate feature 处理链接。
- 尽量不解释业务语义。
- 不在该层主动释放、复制或转换 obmm 资源，除非这是绑定生成所需的最低限度。

初始类型策略：

```rust
#[repr(C)]
pub struct obmm_handle {
    _private: [u8; 0],
}
```

若后续发现 obmm 使用结构体值传递，则必须按头文件补齐 `#[repr(C)]` 布局，并增加 ABI 尺寸、对齐和字段偏移检查。当前不能假定任何结构体布局。

初始链接策略：

- 默认优先动态链接系统库或用户指定路径。
- 通过 feature 区分 `vendored`、`pkg-config`、`static`、`dynamic` 等策略。
- 在不知道 obmm 构建系统前，不默认启用 vendored 构建。

未知项需要后续确认：

- 库文件名。
- include 路径。
- 是否可用 `pkg-config`。
- 静态链接是否需要额外系统库。
- Windows、macOS、Linux 的库名和导出符号差异。

## 安全封装层设计草案

安全 crate `obmm` 应把原始 handle 包装成拥有明确生命周期的 Rust 类型。例如：

```rust
pub struct Obmm {
    raw: NonNull<obmm_sys::obmm_handle>,
}
```

该类型是否实现 `Send`、`Sync` 必须等待 obmm 线程安全契约确认。默认不实现或通过 `PhantomData<Rc<()>>` 保守阻止自动线程安全推导，直到有证据证明可以跨线程使用。

资源释放策略：

- 如果 obmm 提供 `create/destroy` 或 `open/close` 语义，安全层通过 `Drop` 释放资源。
- 如果释放可能失败，`Drop` 中只能做 best-effort 清理；另提供显式 `close` 方法返回错误。
- 如果 obmm 资源必须按顺序释放，安全层类型应表达该顺序，避免用户手动组合裸指针。

当前未知：

- 创建函数名称。
- 销毁函数名称。
- 销毁是否允许空指针。
- 销毁是否可重复调用。
- 销毁是否线程安全。

## 错误处理

建议把 obmm 原始错误表达映射为 Rust enum：

```rust
pub enum Error {
    Unknown,
    NullPointer,
    InvalidArgument,
    Backend(i32),
}
```

这只是占位设计。后续应根据 obmm 的真实错误码、返回值和错误消息 API 调整。

错误处理原则：

- 原始返回码保留在错误类型中，避免丢失调试信息。
- 如果 obmm 提供 `last_error` 或错误字符串函数，需要确认其线程局部性和生命周期。
- 任何 C 字符串转换都必须处理非 UTF-8 数据。
- 不把 panic 穿过 FFI 边界。

当前未知：

- 成功返回值约定。
- 错误码枚举。
- 错误消息所有权。
- 是否存在全局 last-error 状态。

## 字符串与缓冲区

Rust 到 obmm：

- `&str` 转换为 `CString`，显式拒绝内部 NUL。
- 如果 obmm 接受字节缓冲区，应优先暴露 `&[u8]`，避免强制 UTF-8。

obmm 到 Rust：

- 如果返回 `const char *`，必须确认生命周期和编码后再转换。
- 如果返回调用方负责释放的 `char *`，安全层必须绑定对应释放函数。
- 如果使用 “传入 buffer + len” 模式，Rust 层应先查询长度或提供可增长缓冲策略。

当前未知：

- obmm 是否使用 UTF-8。
- 字符串是否以 NUL 结尾。
- 返回字符串是否需要释放。
- buffer 长度单位是字节还是元素。

## 回调与异步

如果 obmm 使用回调：

- Rust 闭包不能直接作为 C 函数指针传入。
- 需要使用 `extern "C"` trampoline 和 `void *user_data`。
- callback 中必须 `catch_unwind`，禁止 panic 穿过 C ABI。
- 必须定义回调注册、注销和资源释放的顺序。

如果 obmm 是异步或事件驱动：

- 先提供同步、安全、低层封装。
- 再根据真实模型决定是否提供 Tokio feature。
- 不在未知线程模型下提前绑定到某个 async runtime。

当前未知：

- 是否存在回调。
- 回调在哪个线程执行。
- 是否允许回调中重入 obmm。
- 是否需要事件循环或外部 poll。

## 全局初始化

如果 obmm 要求全局初始化/清理：

- 使用 `OnceLock` 或等价机制保证初始化只执行一次。
- 谨慎处理全局 cleanup，因为 Rust 程序结束时析构顺序复杂。
- 如果 obmm 的 cleanup 会影响仍存活对象，安全层应通过上下文对象表达生命周期，而不是隐藏在全局 Drop 中。

当前未知：

- 是否需要 `init`/`shutdown`。
- 是否允许多次初始化。
- 初始化是否线程安全。
- 初始化是否依赖环境变量或配置文件。
