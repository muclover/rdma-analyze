## 4. `general-c-ffi.md` 结构与内容约定

> 文件名固定为 **`general-c-ffi.md`**（勿写成 `general-c-fii`）。  
> 标题建议：**《Rust FFI 样本横向对照（P1–P11）》** — 正文说明覆盖全部 11 个样本，含 RDMA。

### 4.1 章节骨架（必达）

| 章节 | 标题（建议） | 内容 |
|------|--------------|------|
| **0** | 范围摘要 | 引用本 `SCOPE.md`；11 项目列表；链到 `rdma-overview.md` |
| **1** | 对照总表 | 行 = P1–P11；列 = D01–D17；单元格 = **短结论 + `→ Pn §x`** |
| **2** | 分维度索引 | 按 D01–D17 分节；每节**索引式**罗列各项目差异（2–5 行/项 + 章节号） |
| **3** | 模式归纳（架构决策向） | 模式 ID（如 M-sys-only、M-sys-safe、M-workspace-adapter…）；**决策问题** → 代表项目 → `→ Pn §10` |
| **4** | 特殊情况 | 见 §4.3；不宜压进总表的样本**单独成节** |
| **5** | 需区分专题 | 见 §4.4；跨项目但需**单独对比**的技术点 |
| **6** | 未覆盖项 | 材料不足、非目标、留给 `rdma-overview` 的细项 |

### 4.2 权威性

- **P1–P11 的权威总表**仅存在于 `general-c-ffi.md` §1。  
- `rdma-overview.md` 可保留 P5–P11 项目级表作便于 RDMA 读者浏览，但**列定义须与 §1 对齐**；冲突时以 `general-c-ffi.md` 为准并注明。  
- 阶段 D 引用对照时：**优先** `general-c-ffi.md` §3 + §4/§5，RDMA 细读再链 `rdma-overview.md`。

### 4.3 「特殊情况」清单（须单独成章 §4）

下列项在 §1 总表中可写「见 §4.x」，在 §4 展开（允许增补，撰写时发现新例外须回写 SCOPE 修订记录）：

| ID | 项目 | 特殊情况（摘要） |
|----|------|------------------|
| S1 | P1 | 同仓双发布 `libz-sys` / `libz-ng-sys`；纯 `-sys` + 重 `build.rs` |
| S2 | P3 | 多 crate（sys / openssl / macros / errors）；宏与 X.509 栈 |
| S3 | P4 | 三层 `zstd-sys` → `zstd-safe` → 顶层；vendored zstd |
| S4 | P7 | 最大 workspace；bnd + `wrapper.c`；**tonic / quinn 上层适配**（零新增 `extern "C"`） |
| S5 | P9 | **不本仓 bindgen**；依赖 crates.io `rdma-sys`（与工作区 P8 非同一交付物） |
| S6 | P10 | **mummy 静态链**；编译期无系统 rdma dev 包 |
| S7 | P11 | safe 依赖 P10；**Extended Verbs + `PostSendGuard`**；运行期仍需系统 rdma-core |
| S8 | P6 | 单 crate 内嵌 bindings；rdmacm 已 bindgen 但 safe 未接 |

### 4.4 「需区分专题」清单（须单独成章 §5）

| ID | 专题 | 涉及项目 | 区分要点 |
|----|------|----------|----------|
| T1 | **inline / 复杂 C 类型** | P1–P4 较轻；P5–P11 为重 | bindgen blocklist、手写 `verbs.rs`、`wrapper.c`、mummy 符号 |
| T2 | **`-sys` 是否独立 crate** | P1,P5,P8,P10 vs P6,P9,P11 | 发布边界、复用 crates.io `-sys` |
| T3 | **链接：`links` / pkg-config / vendored / mummy** | 全体 | 尤其 P1,P4,P5,P10 |
| T4 | **async 集成方式** | P2,P6,P7,P9 vs 其余 | tokio、CQ 驱动、example 实验 |
| T5 | **无硬件 / CI 策略** | P1 systest；P10 build-only；P7 siw/rxe 等 | 阶段 D「测试策略」分支 |
| T6 | **许可证对下游** | P9 GPL-3.0 等 | 与架构「是否 fork 绑定层」相关 |

## 5. 证据深度（裁定：索引级）

| 位置 | 写法 |
|------|------|
| **§1 总表单元格** | 一词/短语结论 + `→ Pn §x`（必要时多条，如 `→ P7 §4.3; §10`） |
| **§2 分维度** | 项目名 + 短句 + 章节索引；避免长段复述单项目分析 |
| **§3 模式** | 模式名 + 决策问题 + 代表项目列表 + `→ §10` 索引 |
| **§4 / §5** | 同上；可附 **路径/符号**（可选，非必须） |

**禁止**：「X 优于 Y」；编造单项目未写的事实。矛盾时以单项目文档为准，在对照中标注「待对齐」。
